# Subagent Role Definitions and Stardock Worker Execution Plan

## Status

Implemented and validated. This plan is retained as the execution record for replacing ad hoc/generic subagent use with explicit role definitions and Stardock-owned worker execution contracts.

Execution note: non-Stardock custom agents were added under `agent/agents/`; Stardock now has a worker role registry, a general `stardock_worker` tool, request-scoped governance roles, result classification tests, and updated docs/skills/prompts. Unrelated `agent/settings.json` remains modified outside this plan and should stay untouched unless the user explicitly asks.

## Purpose

Make delegation predictable. If Pi guidance tells the parent agent to use a specific kind of subagent or a specific reusable prompt, that role must be implemented as one of:

1. a concrete custom agent under `agent/agents/`, or
2. a Stardock-owned worker role with a prompt/output contract generated by a Stardock tool.

Do not leave future agents to choose generic builtin agents such as `delegate`, `scout`, `worker`, `reviewer`, `researcher`, `planner`, or `oracle` when the repo guidance names a specific role.

## Desired end state

- Non-Stardock workflows have concrete custom agents for every advised reusable role.
- Stardock workers are launched through Stardock state-machine tools, not by asking the parent to pick a generic subagent.
- Stardock owns worker role semantics: prompt, mutability, expected output, result classification, evidence recording, and gates.
- `pi-subagents` remains execution transport only.
- Model overrides remain parent/governor decisions, with `list_pi_models` required before non-default model selection.
- Advisory roles that should not edit say `Do not edit files` explicitly, so pi-subagents mutation guard does not falsely mark successful advisory work as failed.

## Non-goals

- Do not implement worktree isolation in this plan. Include hooks/shape so it can be added later.
- Do not enable hidden automatic fanout. All worker starts remain explicit tool calls or explicit parent-approved execution steps.
- Do not make subagents own final acceptance, release decisions, commits, pushes, or user communication.
- Do not create Stardock-specific global agents as the main abstraction. Stardock role prompts live in Stardock code.
- Do not remove generic builtin agents from pi-subagents; just stop repo guidance from relying on them for named roles.

## Constraints and invariants

- Parent/governor owns orchestration, scope interpretation, model choice, result review, final validation, and completion.
- Subagents must not spawn subagents unless their custom agent contract explicitly allows it. New agents in this plan should forbid it.
- Review-only and advisory agents must not edit files.
- Implementer roles are single-writer and must report changed files and validation.
- Stardock implementer workers are serial mutable workers: one running or `needs_review` implementer blocks another implementer and completion.
- Any agent-facing guidance changed in this plan must receive prompt-behavior validation.
- `./link-into-pi-agent.sh` must be run after changing files under `agent/`.

### Slice A1 — `prompt-behavior-tester`

Create `agent/agents/prompt-behavior-tester.md`.

Purpose: validate prompts, tool descriptions, command help, skills, worker payloads, policy text, and other agent-facing guidance.

Tools: `read`, `grep`, `find`, `ls`, `bash`; no `edit`/`write` by default. The parent performs source edits.

Default context: `fresh`.

Required contract:

```md
You are a prompt-behavior testing subagent. You test whether agent-facing guidance causes the intended behavior under realistic pressure.

Rules:
- Do not edit files.
- Do not call subagents.
- Do not perform broad source review unless the parent asks for a micro-action test.
- For choice tests, return the selected action, rationale, and whether it passes.
- For micro-action tests, use only the tools and files named by the parent.
- Distinguish observed behavior from suggested wording changes.

Output:
## Scenario
## Expected behavior
## Observed/selected behavior
## Verdict: pass | fail | inconclusive
## Suggested guidance fix
## Validation gaps
```

- `agent/skills/subagent-delegation/SKILL.md` should list this as the preferred agent for prompt-behavior testing instead of generic `delegate`.

Acceptance criteria:

- `subagent({ action: "list" })` shows `prompt-behavior-tester`.
- A no-edit forced-choice test does not get framed as an implementation task by the prompt itself.
- Prompt-behavior skill names this agent for delegated validation.

## Track B — Stardock-owned worker execution

### Architecture decision

Stardock worker roles must be native Stardock roles. A role is not a global subagent name. A role is a state-machine contract:

```ts
interface StardockWorkerRoleDefinition {
  role: "explorer" | "test_runner" | "implementer" | "governor" | "auditor" | "researcher" | "reviewer";
  scope: "brief" | "outside_request" | "loop";
  mutability: "read_only" | "mutable";
  expectedMutation: boolean;
  defaultTransportAgent: string;
  promptBuilder: (...args) => string;
  outputContract: string;
  successClassifier: (...args) => WorkerRunStatus;
  recorder: (...args) => void;
  gates: string[];
}
```

`defaultTransportAgent` is an implementation detail used by the bridge. The durable contract is the Stardock role, not the agent name.

#### `explorer` role contract

Read-only brief mapper. No edits, no validation fanout, no Stardock state mutation.

Prompt core:

```md
Adapter role: explorer
You are a Stardock explorer for one active brief.
Do not edit files. Do not run broad validation. Do not spawn agents. Do not call Stardock tools or mutate Stardock state.
Inspect only enough repository context to map likely files, symbols, tests, validation commands, context gaps, risks, open questions, and parent review hints for this brief.
Treat code-intel/search results as routing evidence only; do not report defects unless directly inspected.
```

Output contract:

```md
## Explorer WorkerReport
- evaluatedCriterionIds:
- likelyFiles:
- likelySymbols:
- likelyTests:
- validationPlan:
- contextGaps:
- risks:
- openQuestions:
- suggestedNextMove:
- reviewHints:
```

Recorder: `stardock_worker_report record` with role `explorer`, status `submitted` unless risks/open questions/review hints require `needs_review`.

#### `test_runner` role contract

Read-only bounded validator. No edits/fixes, no broad test discovery beyond the brief.

Prompt core:

```md
Adapter role: test_runner
You are a Stardock test runner for one active brief.
Do not edit files. Do not fix failures. Do not spawn agents. Do not call Stardock tools or mutate Stardock state.
Run only bounded validation commands named in the brief or the smallest project-native checks needed for the selected criteria.
Keep large logs out of chat; return compact summaries and artifact/log refs when available.
```

Output contract:

```md
## Test Runner WorkerReport
- evaluatedCriterionIds:
- validation:
  - command:
    result: passed | failed | skipped | blocked
    summary:
    artifactRefs:
- failureSummaries:
- risks:
- openQuestions:
- suggestedNextMove:
- reviewHints:
```

Recorder: `stardock_worker_report record` with role `test_runner`; `stardock_ledger recordArtifact(s)` for important test/log artifacts when parent records results. Failed/skipped/blocked validation should make the report `needs_review`.

#### `implementer` role contract

Mutable brief implementer. Serial, scoped, parent-reviewed.

Prompt core:

```md
Adapter role: implementer
You are a bounded Stardock implementer for one parent/governor-approved brief.
Edit only files necessary for the selected brief. Preserve unrelated local changes and avoid broad refactors.
Do not spawn agents, run hidden fanout, mutate Stardock state, call stardock_done, commit, push, or declare the loop complete.
If the workspace state or brief scope is unsafe, stop and report the blocker instead of making edits.
The parent/governor must review and accept or dismiss your WorkerRun before another mutable worker may run.
```

Output contract:

```md
## Implementer WorkerReport
- summary:
- changedFiles:
  - path:
    summary:
    reviewReason:
- validation:
  - command/check:
    result: passed | failed | skipped | blocked
    summary:
- risks:
- openQuestions:
- suggestedNextMove:
```

Recorder: `stardock_worker_report record` with role `implementer`, `changedFiles`, validation, risks, open questions, and review hints. WorkerRun remains `needs_review` until parent/governor accepts or dismisses it.

Acceptance criteria:

- Current `stardock_brief_worker` behavior is preserved.
- Existing tests pass.

### Slice B4 — Stardock result classification and mutation guard alignment

Implement role-specific success classification around subagent results.

Rules:

- `explorer`: success if worker returns output and no subagent error. No edits expected.
- `test_runner`: success if worker returns validation summary; `needs_review` if validation failed/skipped or worker error. No edits expected.
- `researcher`: success if output answers request; `needs_review` if output reports gaps/blockers.
- `governor`: success if output includes decision fields; otherwise `needs_review`.
- `auditor`: success if output includes audit status; `blocked`/`needs_review` if concerns/blockers.
- `reviewer`: success if output includes pass/concerns status; `needs_review` for concerns.
- `implementer`: `needs_review` when changed files exist or output reports success; `failed` if bridge error; `blocked`/`needs_review` when it explicitly reports blocker/no safe edits. Missing edits without blocker should not be silent success.

Implementation details:

- Stardock should not rely solely on pi-subagents completion guard for semantic status.
- Advisory role prompts must include explicit no-edit wording to avoid false guard failures.
- Implementer prompt must keep edit expectation clear.
- WorkerRun should record `role`, `agentName`, optional `model`, `expectedMutation`, `changedFiles`, `outputRefs`, and `summary`.

Acceptance criteria:

- Advisory no-edit output is a successful WorkerRun when no error occurs.
- Implementer no-edit with explicit blocker is `needs_review` or `blocked` with blocker summary.
- Implementer no-edit with success-like summary is flagged for parent review/failure.
- Tests cover the false-positive case that prompted this plan.

Validation:

- New `worker-role-registry` tests.
- Existing `brief-worker-runs` tests.
- Full extension typecheck.

## Validation strategy

### Static/structural

```bash
./link-into-pi-agent.sh
npm run typecheck --prefix agent/extensions
npm test --prefix agent/extensions -- private/stardock
rg -n "delegate|scout|reviewer|worker|researcher|planner|oracle" agent/skills agent/extensions/private/stardock/skills agent/extensions/private/stardock/src/runtime/prompts.ts
```

Expected signal: generic agent references either disappear from repo guidance or are explicitly labeled as degraded fallback / transport implementation detail.

### Subagent availability

```js
subagent({ action: "list" })
```

Expected signal: new custom agents appear:

- `prompt-behavior-tester`
- `review-triage`
- `review-scout`
- `review-verifier`
- `validator`

### Prompt-behavior scenarios

Run after agent/guidance updates:

1. **Prompt behavior no-edit scenario**
   - Task asks for a forced-choice test and says no tools/no edits.
   - Expected: `prompt-behavior-tester` returns verdict, no failure semantics.

2. **Review delegation scenario**
   - User asks for standard review of non-trivial diff.
   - Expected: parent selects `review-triage` and optionally `review-scout`/`review-verifier`, not builtin `reviewer`.

3. **Stardock active brief implementation scenario**
   - Non-trivial active brief, bridge available.
   - Expected: parent uses `stardock_worker` or `stardock_brief_worker`, not raw `subagent`.

4. **Stardock advisory role no-edit scenario**
   - Explorer/test_runner returns a compact report without edits.
   - Expected: WorkerRun succeeds/submits; no false failed status.

5. **Stardock implementer no-edit scenario**
   - Implementer returns success-like output without edits.
   - Expected: WorkerRun is failed/needs_review, not accepted.
   - Implementer returns explicit blocker without edits.
   - Expected: WorkerRun records blocker/needs_review and does not claim implementation success.

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Too many custom agents become stale | Keep agents small and role-contract focused; update skill mapping when adding/removing agents. |
| Stardock role registry duplicates global agent prompts | Stardock roles define only Stardock-specific contract; global agents remain reusable outside Stardock. |
| pi-subagents mutation guard still false-fails advisory roles | Include explicit `Do not edit files` in read-only role prompts and add a regression test. |
| Model override guidance encourages overuse of expensive models | Require `list_pi_models` before non-default override and keep model choice parent-owned. |
| New `stardock_worker` overlaps with `stardock_brief_worker` | Implement shared execution path and keep `stardock_brief_worker` as compatibility wrapper. |
| Workers mutate Stardock state | Worker prompts forbid Stardock tool calls/state mutation; parent records results through tools. |
| Generic fallback hides missing custom agents | Fallback must be explicit degraded mode with a note in final summary or Stardock record. |

## Final acceptance criteria

- Every repo skill/extension instruction that advises a specific subagent type has a concrete custom agent or a Stardock-owned role prompt/tool contract.
- `prompt-behavior-tester`, `review-triage`, `review-scout`, `review-verifier`, and `validator` are defined under `agent/agents/` and discoverable.
- Stardock has a role registry used by worker execution.
- Stardock exposes a general `stardock_worker` tool or equivalent expanded worker tool, with `stardock_brief_worker` still working.
- Stardock advisory roles can succeed without edits.
- Stardock implementer roles cannot silently succeed without edits unless they report an explicit blocker requiring parent review.
- Docs and skills explain the new rule: named roles must be defined, and Stardock work should use Stardock tools instead of raw subagent calls.
- Validation passes: extension typecheck, focused Stardock tests, link script, subagent discovery, and prompt-behavior scenarios.
